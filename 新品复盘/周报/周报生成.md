# 新品周报看板生成

> 从业务数据生成自包含的新品周报 HTML 看板。零外部依赖（Chart.js CDN 除外），可离线查看，支持交互筛选和一键复制汇报文案。

---

## 一、数据源要求

### 1.1 数据来源

**两种模式**：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **直接 xlsx 读取**（推荐） | Python 脚本直接用 openpyxl 读取源 xlsx，在内存中构建数据块，一次生成 HTML | 5.14+ 周期 |
| JSON 中间层 | 先由 xlsx → JSON（gen_xlsx 脚本），再由 JSON → HTML | 5.13 及之前周期 |

**输入文件**：`周报/新品检查周源数据和PLP数据.xlsx`

### 1.2 各数据块字段定义

#### cum43Data（四三累计主表）

```json
{

  "实际上架日期": "2026-03-04",     // 实际上架日期 YYYY-MM-DD
  "首次出单日期": "2026-03-08",     // 首次出单日期 YYYY-MM-DD
  "4月分析人": "胡煜星",            // 当月分析人
  "品类": "车门系统",               // 产品品类
  "产品拓展": "原开品",             // 拓展类型：原开品/拓展品/组合件
  "4.30-5.6销量": 1,               // 本周销量（数字，0表示未出单）
  "4.30-5.6销售额": 40.0,          // 本周销售额（USD）
  "5.6对手销量": 7,                // 截止日直接竞品出单量

## 二、数据选取与校验规则

### 2.1 数据周期

| 参数 | 值 | 说明 |
|------|-----|------|
| 周期类型 | 周四至周三 | SALES_THU_TO_WED |
| 示例周期 | 4.30 - 5.6 | 周四 = 4/30，周三 = 5/6 |

**字段命名规则**：日期区间作为字段前缀，如 `4.30-5.6销量`、`5.6市占比`、`5.6对手销量`。每周生成时需替换为实际日期。

### 2.2 四三累计数据过滤

cum43Data 必须排除**周期外上架**的新品。过滤条件：
- 实际上架日期 ≤ 周期开始日（周四）
- 即 `实际上架日期 <= 周四日期`

### 2.3 百分比字段强制规则

> **多次事故，绝不再犯。源数据中市占比和PLG费率存储为 0-1 小数，所有输出必须 ×100。**

| 字段 | 列索引 | 源数据示例 | 正确输出 | 错误示例 |
|------|--------|-----------|---------|---------|
| `curMarketShare` / `prevMarketShare` | 51/52 | 0.75 | `75%` 或 `75` | `0.75%` |
| `plgFee` | 130 | 0.02 | `2%` 或 `2.0` | `0.02%` |

**检查清单（每次修改提取代码后必须确认）：**
- [ ] `curMarketShare` / `prevMarketShare` = `round(num(col) * 100, 1)`
- [ ] `plgFee` = `round(num(col) * 100, 1)`
- [ ] 筛选阈值匹配 ×100 后的量纲（`>= 75` / `>= 50` / `< 50`）
- [ ] HTML 生成脚本和 XLSX 生成脚本**两边都要检查**

### 2.4 数据校验清单

生成前必须验证：

1. **28 个 key 全部存在**：缺少任何 key 报错终止
2. **cum43Data 非空**：至少 1 条记录
3. **SKU 唯一**：cum43Data 中 SKU 不重复
4. **环比一致性**：prevWeekKpi.prevTotalSku 与上周实际 SKU 数匹配
5. **数组长度合理**：
   - plpDetailData 通常 40-60 条
   - plgRecords 与 cum43Data 条数一致
   - expandTypeData 通常 2-3 条
   - plpAnalysts 通常 5-7 条
6. **金额为 USD**：销售额、花费等字段单位为美元，不做货币转换
7. **百分比格式**：市占比字段为 0-100 的数字（不是 0-1 小数）

### 2.4 核心业务规则

| 规则 | 说明 |
|------|------|
| 低占比新品判定 | 市场份额 < 0.75 **且** 存在竞品订单（对手销量 > 0） |
| 8日出单 Y/N | 上架后 8 天内出单 = Y，超过 8 天 = N |
| 出单均价 | 销售额 / 销量，销量为 0 时为 null |
| 在售均价 | 有效在售价简单平均（过滤空值、0、负数、>=10000） |
| 环比计算 | 只能在同周期类型内比较（周四至周三 vs 周四至周三） |
| PLP 广告 | Amazon Product Listing Promotion，分析人手动设置 |
| PLG 广告 | Product Listing Group，通过费率标识（如 "2.0%"） |
| ACOS | 广告销售成本比 = spend / adRevenue（广告花费 / 广告销售额） |
| ACOAS | 全盘销售成本比 = spend / totalRevenue（广告花费 / 总销售额）。汇总时需按 SKU 去重避免重复累计 |

}

- 每个变量以分号结尾

### 7.4 关键注意事项

1. **禁止在 Python f-string 中使用裸 `\n`**：必须使用 `\\n` 转义，否则生成的 JS 会出现多行字符串语法错误
2. **反引号模板字面量**：JS 中多行拼接优先使用反引号模板字符串，`\n` 写成 `\\n`
3. **括号匹配**：复杂表达式（如嵌套的 `filter` + `Math.round`）必须确保括号平衡，用 Node.js `vm.Script` 验证
4. **美元符号转义**：Python f-string 中 `$` 无需转义，但 `{` 和 `}` 需要双写为 `{{` 和 `}}`
5. **图表延迟初始化**：Chart.js 图表在 Tab1 首次可见时才创建，避免隐藏 canvas 导致尺寸为 0
6. **PLG 匹配**：在 Tab4 筛选中，通过 `plgRecords` 构建 `plgMap`（以 SKU 为 key），实现 PLP/PLG 状态的快速查找
7. **ACOAS 数据格式兼容**：corrected_data.json 中 ACOAS 存在两种格式——字符串百分比（如 `"3.92%"`，来自 plpTotal/plpPrevTotal/plpAnalysts/plpCategories）和数值小数（如 `0.0423`，来自 plpSummaryData/plpDetailData）。**必须使用 `acoasPct()` 函数统一处理**，禁止直接对 acoas 字段做 `*100` 运算（字符串乘法会产生 NaN）
8. **ACOAS SKU 去重计算**：plpDetailData 中同一 SKU 可能出现在多个广告活动中（如 smart~xauto 和 smart~via），每行都带相同的 `totalRevenue`（SKU 总销售额，非广告销售额）。汇总 ACOAS 时**必须按 SKU 维度去重**，避免 totalRevenue 被重复累计导致 ACOAS 偏低。具体规则：
   - **概览核心指标 / 分析人维度 / 品线维度**：从 plpDetailData 按 SKU 去重构建总销售额映射，用 `总花费 / 去重总销售额` 计算 ACOAS
   - **广告明细表格**：保持广告活动维度逐行展示，每行 ACOAS = spend / totalRevenue（行级计算无需去重）
   - **上周 ACOAS**：沿用 plpPrevTotal.acoas 原值（无明细可去重）

### 8.2 直接 xlsx 模式流程

```
1. openpyxl 读取 周报/新品检查周源数据和PLP数据.xlsx
2. 读取 Sheet"四三数据累计"（137列主表）+ Sheet"PLP明细"
3. 在 Python 内存中构建 23 个 JS 数据块（list/dict → json.dumps）
4. 过滤规则：实际上架日期 ≤ 截止日期，排除未来上架的 SKU

### 8.3 Python 生成脚本模板（直接 xlsx 模式）

```python
# -*- coding: utf-8 -*-
import json, os
from openpyxl import load_workbook

SRC = "周报/新品检查周源数据和PLP数据.xlsx"
OUTPUT = "新品板块_5.14-5.20.html"

wb = load_workbook(SRC, data_only=True)
ws43 = wb["四三数据累计"]
wsPLP = wb["PLP明细"]

# --- 构建 23 个数据块 ---
# cum43Data: 遍历 ws43 行（排除表头），提取 137 列
# lowShareData: 按低占比规则过滤
# plpDetailData: 遍历 wsPLP 行（排除汇总行：SKU='广告链接数量：'）
# plpTotal/plpPrevTotal/plpAnalysts/plpCategories/plpExpandTypes: 从 PLP 聚合
# ...

DATA_KEYS = [
    "cum43Data", "cum43Stats", "lowShareData", "expandTypeData", "timelinessData",
    "hasCompetitorUnsold", "plpTotal", "plpPrevTotal", "plpCategories", "plpExpandTypes",
    "plpAnalysts", "unsoldNoCompetitor", "prevWeekKpi", "plgStats",
    "prevWeekCategories", "prevWeekAnalysts", "plgRecords",
    "categoryRevenueData", "analystRevenueData", "plpSummaryData", "plpDetailData",
    "prevWeekPlpCategories", "prevWeekPlpAnalysts",  # 23 个 key
]

for k in DATA_KEYS:
    if k not in locals():
        raise ValueError(f"Missing key: {k}")

js_vars = "\n".join(
    f"const {k} = {json.dumps(v, ensure_ascii=False, indent=0)};"
    for k, v in [(k, locals()[k]) for k in DATA_KEYS]
)

# 构建 HTML（CSS 独立模板 + body 独立模板 + JS 独立模板）
# 注意：Python f-string 中 { } 双写为 {{ }}，字符串 \n 写为 \\n

html = f'''<!DOCTYPE html>
<html lang="zh-CN">
<head>
<meta charset="UTF-8">
<title>新品板块_5.14-5.20</title>
<script src="https://cdn.jsdelivr.net/npm/chart.js@4.4.0/dist/chart.umd.min.js"></script>
{CSS_TEMPLATE}
</head>
<body>
{BODY_TEMPLATE}
<script>
{js_vars}
{JS_TEMPLATE}
</script>
</body>
</html>'''

with open(OUTPUT, "w", encoding="utf-8") as f:
    f.write(html)
```

> **模板分离**：对于大型脚本（1000+ 行），建议将 CSS、HTML body、JS 分别抽成独立模板变量或文件，避免单个 f-string 过长。

## 十、常见问题与排错

### 10.1 JS 语法错误

| 症状 | 原因 | 修复 |
|------|------|------|
| `Unexpected token ')'` | Python f-string 中裸 `\n` 导致 JS 多行字符串 | 改用反引号 + `\\n` |
| `switchTab is not defined` | 上游 JS 语法错误导致整段脚本解析失败 | 用 vm.Script 定位具体行 |
| `Unexpected token '}'` | 括号不匹配 | 检查嵌套 filter/Math.round 表达式 |

### 10.2 数据显示异常

| 症状 | 原因 | 修复 |
|------|------|------|
| KPI 全部显示 "-" | 数据未正确嵌入 | 检查 JSON 序列化是否含中文转义 |
| 图表不渲染 | Canvas 在隐藏 Tab 中初始化 | 使用延迟初始化 `setTimeout(initCharts1, 100)` |
| 筛选器无选项 | 分析人/品类未动态填充 | 检查 `cum43Data` 中字段名是否匹配 |
| ACOAS 显示 NaN% | 对字符串格式的 acoas（如 `"3.92%"`）直接做 `*100` 运算 | 使用 `acoasPct()` 函数统一处理，自动判断字符串/数值格式 |
| ACOAS 值偏低 | plpDetailData 中同一 SKU 多广告活动导致 totalRevenue 重复累计 | 汇总层按 SKU 去重计算 ACOAS（见第 7.4 节第 8 条） |
| 浏览器刷新后数据未更新 | 浏览器缓存旧文件 | 修改 `<title>` 加版本号（如 v3.2），用户 Ctrl+Shift+R 强制刷新 |

### 10.3 编码问题

- **禁止**使用 PowerShell 直接操作含中文的 HTML 文件（会损坏 UTF-8 编码）
- 始终使用 Python 脚本进行文件读写，确保 `encoding="utf-8"`
