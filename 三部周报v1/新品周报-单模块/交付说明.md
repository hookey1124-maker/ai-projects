## 2. 页面是如何设计的

### 整体架构：4-Tab 布局

页面采用四个标签页组织全部内容，遵循主项目 UI 风格（浅灰背景、白色卡片、蓝色主强调色）：

#### Tab 1：总盘概览
- **KPI 卡片区**：在售 SKU 总数、8 日内出单(Y)、8 日外出单(N)、未出单合计、已出单合计、总销量、总销售额、低占比新品数
- **品线维度表**：7 个品线的 SKU 数、新上架数、销量、销售额（含环比）
- **分析人维度表**：6 位分析人的 SKU 数、新上架数、销量、销售额（含环比）
- **拓展类型表**：原开品/拓展品的出单率、销量对比
- **分析及时率表**：各分析人的及时分析数、及时率、8 日内无分析、超 7 日未分析
- **新品出单情况**：有对手口径的 Y/N/未出单分布，按分析人/品线维度交叉分析

#### Tab 2：低占比分析
- **低占比新品概览**：按品类/出单状态分布的图表
- **低占比新品明细表**：39 条记录，支持按市场状态/分析人/品类/8 日出单筛选
- **新品未出单原因分析**：有对手未出单 3 个 + 无对手未出单 27 个，按分析人/品线维度交叉（竞争无优势/无市场双口径）
- **未出单原因分布图**：饼图展示原因占比

#### Tab 3：广告追踪（PLP 复盘）
- **PLP 总览**：42 个 SKU、119,686 曝光、$272.56 花费、ROAS 6.44、ACOS 15.54%、ACOAS 3.92%
- **核心指标卡片**：ROAS/CVR/CTR/CPC/CPA/ACOS/ACOAS
- **分析人维度 PLP 表**：6 位分析人的广告投放效果（含 spend/exposure/clicks）
- **品线维度 PLP 表**：7 个品线的广告投放效果
- **广告条件筛选器**：支持按 PLG 条件过滤 cum43 数据
- **四三累计数据表**：101 条 SKU 明细，支持 8 维筛选（市场状态/分析人/品类/拓展类型/8 日出单/广告条件/市占比/竞争情况）

#### Tab 4：汇报输出
- **KPI 概览**：6 个核心指标卡片（含风险状态标签）
- **风险预警**：基于 Rules 引擎动态生成
- **主要发现**：基于数据动态分析（出单率趋势、未出单原因、及时率变化、品类异常）
- **下周动作**：基于异常规则的建议 + 通用优化建议
- **可复制周报文案**：一键复制 5 段式汇报文本（核心指标、风险预警、主要发现、品类维度、下周动作）

### UI 风格
- 使用主项目公共组件：`ModulePageLayout`、`KpiCard`、`SectionCard`、`StatusTag`、`EmptyState`
- 配色语义化：绿色=正常、蓝色=核心指标、琥珀色=警示、红色=风险
- 筛选器支持独立标签显示 + 一键重置
- 图表使用 Recharts（圆环图、柱状图）

## 4. Adapter 做了什么

`newProductStatusAdapter.ts` 负责数据层工作，主要功能：

### 类型定义
定义了完整的 TypeScript 类型体系：
- `NewProductStatusData`：模块顶层数据结构
- `OverallData`/`OverallKPI`/`TimelinessMetrics`/`SalesSituationMetrics`：总体数据
- `CategoryMetrics`/`AnalystMetrics`/`ExpandTypeMetrics`：维度数据
- `LowShareRecord`/`PLPData`/`PLPAdMetrics`/`PLGData`：明细数据
- `SalesSituationDetail`/`UnsoldReasonData`：出单/未出单数据
- `NewProductAnomaly`：异常数据

### 数据加载与转换
- `loadCorrectedNewProductData()`：从 `corrected_data.json` 加载修正数据
- `parseNewProductStatusData()`：将原始 JSON 转换为模块所需的类型化数据
- `mockNewProductStatusData`：导出 mock 数据供开发和测试使用
- 数据转换包括：字段名映射、数值类型转换、缺失值处理、环比计算

### 数据聚合
- 从 101 条 cum43 明细聚合出 7 个品类维度和 6 个分析人维度的统计
- 从 lowShareRecords 聚合出各品线/分析人的低占比分布
- 从 PLP 数据聚合出分析人/品线维度的广告指标

## 5. Rules 做了什么

`newProductStatusRules.ts` 实现了 5 条异常检测规则，输出 `NewProductAnomaly[]`：

| 规则 ID | 描述 | 触发条件 | 严重级别 |
|---------|------|---------|---------|
| `new_product_low_share_high_risk` | 低占比高风险 | 有竞品的低占比新品数 > 0 | 高 |
| `new_product_no_sales` | 有对手未出单 | 有对手新品中存在未出单 | 中 |
| `new_product_ramp_slow` | 爬坡缓慢 | 8 日外出单占比 > 50% | 低 |
| `new_product_analysis_delay` | 分析延迟 | 总体分析及时率 < 60% | 中 |
| `new_product_competitor_pressure` | 竞品压力 | 8 日内出单率较上期下降 > 10pp | 低 |

辅助函数：
- `getOverallRiskLevel()`：综合所有异常返回总体风险等级（high/medium/low/unknown）
- `newProductStatusRuleIds`：规则 ID 常量数组

## 6. Summary 做了什么

`newProductStatusSummary.ts` 负责生成周报文案汇总，输出 `ModuleStatusSummary`：

### 核心指标（coreMetrics）
提取 4 个关键指标：累计 SKU、总销量、出单率、分析及时率

### 主要发现（mainFindings）
基于数据动态分析，自动生成发现文案：
- 出单率趋势（较上期上升/下降/持平）
- 未出单情况（有对手未出单数量及原因）
- 分析及时率变化
- 总销量环比变化

### 下周动作（nextActions）
基于异常规则的 recommendation 自动生成建议：
- 若存在异常 → 输出各异常的改进建议
- 若无异常 → 输出通用观察建议

### 风险状态（riskLevel + currentStatus）
调用 `getOverallRiskLevel()` 得出风险等级，映射为中文状态：
- high → 高风险
- medium → 中风险
- low → 低风险
- unknown → 待接入

### 预定义
- `emptyNewProductStatusSummary`：数据未接入时的空状态
- `ReportText`：汇报文案类型定义（full/summary）
